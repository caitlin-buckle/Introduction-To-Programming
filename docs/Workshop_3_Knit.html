How to Source Code from R Script in Rmd (without having to copy and paste)

To source the code in our workshop script we can use the source() function in R. This will execute the code in the specified file and make any functions or variables defined in that file available in our current R session.

EXAMPLE CODE: source (”../code/Module_1.R”)

the “../” will instruct R to go ‘up’ one folder (or out of one folder) and into the next required folder “/code/” in this example.

NOTE: the source() function will not work unless the code in your script has no issues or errors. If the R-script file is complete, the source should run.

source("Module_1.R") #The R-script file is in the same folder on my computer as the R project file, therefore I do not need to specify having to exit a folder and into another one (therefore don't need to include "../" in my code). 
print(plot) # in R-script, the ggplot will appear in the plot window automatically, but if you don't have it specified for R to print the plot, you will not see anything when you source the plot from your R-script in Rmd.

Section 3.8: Workflow with Git and GitHub

Using version control (Git) changes our workflow a bit. E.g., Edit –> Save –> Stage –> Commit –> Pull –> Push –> etc.

So far, we have conducted all of these tasks expect pulling. We need to pull when we are collaborating with others on a repo and there are changes we need to incorporate onto our own version from the remote.

Could create a README file on your local version and push it to the remote (how you would normally manage this sort of task), for now can just click on “Add a README” in Github, with a short statement relating to what the repo is all about. README file saved to repo by selecting green “Commit changes” button.

Summary of Git/GitHub Terminology

Pull = pulling a repository is like syncing your local project from the remote server (GitHub) onto your computer, and combining it with whatever you have on your computer. This is vital to do if someone else has done some work on it, or you’re working on the same file from several computers. Make sure you save, stage, and commit your own work first though, or it may get overwritten

Stage = Git Stage prepares your files to Commit. You can decide which files you want to commit by staging just those files

Commit = Git Commit is that function that saves this ‘version’ of your repo into your repositories version history. However, it is only saved locally at this point (not on your remote repository on the GitHub website). Make sure you include a meaningful human-readable Commit message of what you’ve done to your repo.

Push = Once you are happy with your work and have Committed to Git locally, Push is the function that sends your changes to your remote server and synchronises with that version. If you want others to see the work you’ve done, or you want to save it properly as a backup, or access your work from a different computer, then you need to push your repo to GitHub. Remember, you can Commit without Pushing, which can be useful if you want to track files locally before sending to GitHub.

If you want to see your Commit history, just click on the little clock to see all of your versions and you can go back at any time you like.

After committing changes on GitHub (to add README file), select Pull on R Project and the README.md file will appear in your project (need to tell Git to obtain information from Github and vice versa).

Committing your work to GitHub should be done often, and Git can’t track a file without staging and committing it. Remember to follow the workflow of Edit –> Save –> Stage –> Commit –> Pull –> Push.

NOTE: You will only see files/folders in your Git pane if it is uncommitted. If there is nothing there, everything has already been committed!
